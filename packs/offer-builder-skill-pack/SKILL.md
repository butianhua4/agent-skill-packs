# Offer Builder Skill

## Purpose

Convert a public market signal, buyer pain, demo, or reusable skill pack into a
clear sellable offer. Use this when an operator needs a Fiverr gig, custom offer,
domestic platform listing, productized service package, or proof-backed launch
plan.

## Inputs

Accept one or more of:

- Market opportunity radar output
- Buyer pain notes or redacted inquiry summaries
- Existing GitHub repo, demo, release, or proof asset
- Skill-pack README, delivery template, or test case
- Target platform, target buyer, and preferred price band

Reject or ask for redaction when inputs contain:

- Passwords, tokens, cookies, secret values, payout details, KYC material, or ID
  documents
- Private buyer data, private repositories, unreleased private code, or sensitive
  screenshots
- Requests to guarantee income, ranking, platform approval, or specific order
  volume

## Workflow

1. Define the buyer:
   - Role
   - Current pain
   - Trigger event
   - Why they would pay now
2. Define the outcome:
   - Diagnosis
   - Fix plan
   - Implementation
   - Template/skill pack
   - Reporting or handoff
3. Build the package ladder:
   - Entry offer: low-risk diagnosis or audit
   - Standard offer: scoped repair plan or implementation
   - Premium offer: multi-workflow rescue, custom pack, or rollout support
4. Set scope boundaries:
   - Included
   - Excluded
   - Buyer must provide
   - User-only actions
5. Create proof requirements:
   - Demo screenshot
   - Sample report
   - Before/after example
   - GitHub link
   - Delivery template
6. Write launch copy:
   - Title
   - One-line offer
   - Package descriptions
   - Buyer requirements
   - FAQ
   - Safe custom-offer message
7. Recommend the next 30-minute action.

## Output

Return a concise Markdown offer plan:

- Offer decision
- Buyer and pain
- Package ladder table
- Title options
- Buyer requirements
- Delivery scope
- Proof asset checklist
- FAQ / objection replies
- Next 30-minute action

Use this table:

| Tier | Buyer problem | Deliverable | Price band | Timeline | Upgrade path |
| --- | --- | --- | --- | --- | --- |

## Boundaries

- Do not promise revenue, order volume, platform rank, virality, or approval.
- Do not create offers that require account login, payout, wallet, tax, payment,
  KYC, identity verification, or private customer data.
- Do not recommend broad retainers before a small proof offer exists.
- Prefer fixed-scope, proof-backed offers with clear stop conditions.
- Ask the user to confirm before publishing a real gig, sending a custom offer,
  or changing a platform listing.
